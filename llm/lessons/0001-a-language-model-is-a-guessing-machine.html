A Language Model Is a Guessing Machine

By the end of this page you will have built a working language model. No PyTorch, no maths, no training. Just counting — and it will generate text.

You said you use these things every day and can't explain what's inside. So let's remove the mystery in one move, on the first page, before any machinery arrives. Here is the entire definition, and it is not a simplification:

A language model is a function that takes some text and returns a probability for every token that could come next.

That's it. That is what GPT-4 is. That is what GLM-4.5 is. Not "an AI that understands language" — a function from a context to a list of numbers that sum to 1. Every impressive thing you have seen a chatbot do is that function, called over and over, each time on its own previous output.

Why start here. Almost everything that confuses people about LLMs dissolves once this is concrete. Hallucination, temperature, streaming, context limits — all of them are properties of this function and the loop around it, not of some further magic.

The picture to keep

Ask a model to continue the cat sat on the. It does not "choose" a word. It returns a number for every token it knows — this is the picture you'll see in every lesson from here on:

mat61%
floor18%
sofa9%
table5%
roof2%

Illustrative numbers, to fix the shape of the idea. Real ones are coming — you're about to compute them yourself.

The remaining tokens in the vocabulary — all fifty thousand of them, for GPT-2, whose config lists vocab_size: 50257 — share the leftover probability, most of them a vanishingly small slice. The output is always the same shape: one number per token, (V,) where V is the vocabulary size.

Shapes, always. From here on, every tensor gets its shape and the meaning of each axis. It is the habit that separates people who can read model code from people who can only run it. (V,) means "a flat list of V numbers."

Build one by counting

Where do those numbers come from? In a real model, from billions of learned parameters. But you can get them from something you already understand completely: counting.

Take a tiny corpus. Walk through it and tally, for each token, which token came next. That tally is a language model — the oldest kind, called a bigram model, because it looks at one token of context to guess the next.

Those four percentages are not an illustration. They are the real distribution of that corpus after the token the, and the widget below will show you exactly those numbers.

Now drive it

Below is that model, built live from that corpus. Pick a context, read the distribution, then generate. Two things to actually do — the point is the doing, not the reading:

  1. Before clicking, predict which token wins from model. Then look.
  2. Switch to Greedy and generate ten tokens. Watch what goes wrong.

Drill 1 · Interactive

Your first language model

27 tokens of corpus, a 12-token vocabulary, 20 learned numbers. It is a real language model by the definition above — no asterisk.

Two discoveries are waiting in there, and they are the whole lesson.

One: sampling is why it isn't deterministic

The model gives the same distribution every time. What differs is how you use it. Sampling rolls a die weighted by those probabilities — so the becomes model 38% of the time and text 13% of the time. Run it twice, get two texts.

Greedy decoding always takes the top bar. It is deterministic, and it is worse. Try it: from the, greedy produces

the model again . the model again . the model again . the

It locks into a cycle, forever. Of course it does — the same context always yields the same choice, so the instant the path revisits a token it has already emitted, it is trapped. The randomness you might have assumed was a defect is what keeps generation alive.1

1. This is why real decoders ship a temperature knob rather than always taking the argmax, and why setting temperature to 0 makes a chatbot repeat itself. Same phenomenon, bigger model.

Two: the loop is the whole of generation

Generating text is this, and nothing else:

Context"the"
→
Model(V,) probs
→
Pick one"model"
→
Append"the model"
↺

Feed the output back in as input. Repeat. That is autoregressive generation, and it is why a chatbot's answer arrives word by word rather than all at once: each token is a separate call that could not have happened before the one in front of it existed.

Drill 2 · Retrieval

What does a language model output?

What your model cannot do

Your bigram sees exactly one token of context. Ask it to continue the and it cannot know whether the sentence began the model reads… or the context feeds…. Everything before the last token is gone.

So it can never learn that token tends to follow next only when you're talking about tokens. It has no memory, and no way to notice that model and context are the kind of word that behaves alike. It stores a separate, unrelated number for every pair it ever saw.

Fixing those two limits — reach further back, and let similar tokens share what they learn — is, honestly, the rest of this course. Attention is the answer to the first. Embeddings are the answer to the second.

With one token of context, what can a bigram never do?

The gap, in numbers

You wanted to know how far this is from GLM. Let's be exact rather than vague, now and every time. Your model holds 20 numbers. Here is the same axis, all the way up — logarithmic, because on a linear axis your model would be less than one pixel:

Yours (today)20
TinyStories~10M
GPT-2 small124M
nanochat d20560M
GLM-4.5 active32B
GLM-4.5 total355B

Parameters, log₁₀ scale — each full bar-width spans ten orders of magnitude. GLM-4.5 is 355B total but only 32B active per token: it is a mixture of experts, so most of it sits idle on any given word.

Two things are true at once, and holding both is the discipline of this whole workspace.

The gap is enormous and it is not mysterious. GLM-4.5 saw 23 trillion tokens; you saw 27. Its training run cost roughly 3 million H100-hours by our own estimate — and you will be able to derive that number yourself before this course ends. But nothing on that chart is doing something categorically different from what you just did. They are all returning a probability for every next token.

And small is not the same as useless. Microsoft Research's TinyStories showed that models under 10M parameters write fluent, coherent, grammatical English — if the data is narrow enough. Meanwhile Karpathy's own $100 nanochat tier (560M parameters, four hours on eight H100s) produces something he describes as "like talking to a kindergartener."2 Scale buys breadth. It does not buy the definition — you already have that.

2. Karpathy, nanochat Discussion #1. Worth internalising: every nanochat tier needs an 8×H100 node. None of them run on your Mac. We will reproduce that pipeline stage for stage, at a scale that fits here — and the babble it produces will be a success, not a failure.
The one trap

When you train your own model in a few lessons' time, it will produce nonsense, and some part of you will read that as having done it wrong. You didn't. A model this size should babble. Judge your model against its parameter count and its token count — never against a chatbot that cost eight figures.

What you now know

  • A language model maps a context to a probability over every next token. Shape (V,).
  • You can build one by counting pairs and dividing. You did.
  • Generation is a loop: predict, pick one, append, repeat.
  • Sampling gives variety; greedy gives loops.
  • One token of context is the bigram's ceiling — and the reason attention exists.

Next we take the word "token" seriously. You have been assuming it means "word." It doesn't, and the difference is the first real stage of the pipeline you're here to build.
