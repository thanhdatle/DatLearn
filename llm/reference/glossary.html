The vocabulary

Defined once, so every lesson uses the same words for the same things. Terms marked ahead haven't been taught yet — they're here so that when you meet them in a paper, you have somewhere to look. Prints cleanly (⌘/Ctrl-P).

The core loop

Token
The atom a language model reads and writes. Not quite a word: real tokenisers split tokenising into pieces, and punctuation is usually its own token. A model never sees letters — only token IDs, which are just integers indexing a vocabulary.
Vocabulary size = V
The fixed set of tokens a model knows, each with an integer ID. Everything the model can ever say is a sequence drawn from this set. GPT-2's vocabulary is 50,257 tokens; the toy model in Lesson 01 has 12.
Tokeniser
The (non-neural) program that converts text ↔ token IDs. It is trained separately from the model, before the model exists, and its choices are permanent — you cannot change the vocabulary without retraining everything downstream.
Corpus
The body of text a model learns from. GLM-4.5 was trained on 23 trillion tokens (tech report). Lesson 01's corpus is 27 tokens. The distance between those two numbers is most of what separates the two models.
Context
The tokens the model is allowed to look at when predicting the next one. A bigram model's context is exactly one token. GPT-2's was 1,024; GLM-4.6's is 200,000 (model card). Also called the context window.
Language model
A function from a context to a probability distribution over the next token. That is the complete definition. Everything else — transformers, attention, trillions of parameters — is machinery for making that function good.
Probability distribution
A list of numbers, one per token in the vocabulary, each ≥ 0, summing to exactly 1. The model's entire answer at every step. Shape (V,) for one prediction.
Autoregressive
Generating one token at a time, then feeding your own output back in as context. The reason a chatbot streams word by word: each word is a fresh forward pass that depends on every word before it. Also why generation cannot be parallelised the way training can.
Greedy decoding
Always take the single highest-probability token. Deterministic, and it tends to fall into loops — the same context always produces the same choice, so any cycle repeats forever.
Sampling
Draw the next token randomly, in proportion to its probability. Run twice, get two different texts. This — not the network — is why the model is not deterministic.
Temperature ahead
A knob that sharpens or flattens the distribution before sampling. Below 1 makes the model more confident and repetitive; above 1 makes it more surprising and less coherent. At 0 it becomes greedy.

Tokenisation

BPE — byte-pair encoding
The algorithm that builds a vocabulary: start from single characters, repeatedly find the most frequent adjacent pair, and merge it into a new token. Stop when the vocabulary reaches the size you want. Every merge is learned from the corpus by counting — no neural network is involved. Originally a text-compression algorithm; adapted to language by Sennrich, Haddow & Birch, Neural Machine Translation of Rare Words with Subword Units (arXiv 2015, ACL 2016).
Merge rules
The ordered list BPE produces. Order matters: rule 1 is applied everywhere before rule 2 is considered. Encoding new text means replaying these rules in the same order. This list is the tokeniser.
Subword
A token smaller than a word and bigger than a character. Subwords are the compromise that lets a fixed vocabulary represent any word ever written: common words become one token, rare ones get spelled out from pieces.
Out-of-vocabulary (OOV)
A word the tokeniser has no token for. Word-level tokenisers break on these; subword and byte-level tokenisers cannot, because they can always fall back to smaller pieces. This is the entire reason BPE won.

Models and their parts

Parameter
One number the model learned from data, rather than one you chose. Count them and you have the model's size. Lesson 01's bigram model has 20. GLM-4.5 has 355 billion. Same job; ten orders of magnitude apart.
n-gram model
A language model built by counting: how often did token B follow context A? Normalise the counts and you have a distribution. Bigram = 1 token of context, trigram = 2. The oldest kind of language model, and still the clearest.
Embedding ahead
A lookup table mapping each token ID to a learned vector, shape (V, C). Where a bigram stores a separate count for every token pair, an embedding lets similar tokens share what they learn. This is the single idea that lets models generalise beyond what they saw.
Logits ahead
The model's raw, unnormalised scores over the vocabulary — any real number, positive or negative. Shape (V,). Softmax turns them into probabilities.
Softmax ahead
Exponentiate every logit, then divide by the total. Produces positive numbers that sum to 1, i.e. a distribution. The exponential is what makes a slightly higher score into a much higher probability.
Loss ahead
A single number measuring how surprised the model was by the token that actually came next. Low loss = the model expected it. Training is nothing but pushing this number down.
Attention ahead
The mechanism that lets a token's prediction depend on any earlier token in the context, not just the previous one — and lets the model learn which earlier tokens matter. The contribution of Vaswani et al., 2017, and the reason a bigram's one-token memory is not a permanent ceiling.
Transformer ahead
The architecture stacking attention and small feed-forward networks into layers. Every model in this glossary — GPT-2, GLM, Qwen — is a transformer with different dimensions and a handful of refinements.

Scale — the words for the gap

Mixture of Experts (MoE)
A layer holding many parallel feed-forward "experts", of which only a few run for any given token. Lets a model own enormous total parameters while paying for only a few per token. GLM-4.5 has 1 shared + 128 routed experts and uses 8 routed experts per token (LMSYS).
Total vs active parameters
In an MoE, total is what you must store; active is what actually computes for each token. GLM-4.5 is 355B total / 32B active. Cost and speed track the active count; memory tracks the total. Quoting only one number is how people mislead you.
FLOPs, and C ≈ 6ND
The total arithmetic a training run performs. The standard estimate (Kaplan et al., 2020) is 6 × active parameters × training tokens. For GLM-4.5: 6 × 32e9 × 23e12 ≈ 4.4 × 10²⁴ FLOPs. Learn this formula — it converts any model's headline numbers into an honest cost.
MFU — model FLOPs utilisation
The fraction of a chip's peak arithmetic rate a real training run achieves. Large runs land around 35–45%. An H100 peaks near 989 TFLOP/s in BF16, so it really delivers ~350–450. Dividing C by that gives GLM-4.5 ≈ 3 million H100-hours — roughly 120 days on a thousand GPUs. Zhipu has never published a cost; this is an estimate from their published sizes, and you should be able to derive it yourself.
Scaling laws
Loss falls as a smooth power law in parameters, data, and compute — predictably enough to plan a run before starting it (Kaplan 2020). Chinchilla refined the compute-optimal ratio to roughly 20 training tokens per parameter.

Training stages — the pipeline we're reproducing

Pretraining
Predict the next token over a very large, general corpus. Produces a base model: fluent, knowledgeable, and useless as an assistant — it completes text rather than answering. This is where essentially all the compute goes.
Midtraining ahead
A shorter stage on curated, higher-quality data (code, maths, cleaned web) that teaches formats and conversational structure before alignment proper. Named as its own stage in nanochat.
SFT — supervised fine-tuning ahead
Train on curated (prompt, good-response) pairs so the base model learns to answer rather than continue. The first step of the InstructGPT recipe.
RLHF / DPO ahead
Optimise against human preference rather than a fixed target. RLHF trains a reward model and runs RL against it; DPO reaches a similar place without the RL.
Distillation ahead
Train a small student to imitate a large teacher. The one honest route by which something laptop-sized can beat something enormous — but only on a narrow task. Google's Distilling Step-by-Step got a 770M student past 540B PaLM on-task, ~700× smaller.
LoRA / QLoRA ahead
Freeze a pretrained model and train small low-rank adapter matrices instead of all weights (LoRA). QLoRA adds 4-bit quantisation, putting a 7B fine-tune inside ~5GB of VRAM.
